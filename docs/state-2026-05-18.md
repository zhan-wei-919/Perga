## 1. 总体目标

Perga 是一个**以 Web 前端为主**、Rust 后端接管 PTY 和终端协议解析的新形态终端。终端字节流由后端 alacritty 内核解析,前端只消费规范化后的协议事件(`Init` / `Patch` / `Exited`)。

最终形态(目标态):

- 桌面 Tauri app:本地 PTY + russh,所有 auth 本地。
- iPad / Android Tauri app:russh 直连远程机器(平板沙箱不允许 spawn ssh 子进程)。
- 中间形态(当前开发态):**纯 Web,前端浏览器 ↔ 本机 Rust server**,绕开 Tauri 调试摩擦。

用户给出的 5 个产品需求:

1. 好看,自定义布局。
2. 多 SSH 会话。
3. **未来重要场景是 iPad / Android 远程 SSH 连一台机器**。这反过来要求 SSH 走 russh,不走 `ssh` 子进程。
4. 分块布局 = **tabs(顶部)+ panes(tab 内分屏)+ Warp 风格命令 block(完成的命令收成 DOM 块,但当前行仍是 inline 终端体验,不是固定底部输入框)**。
5. 自定义缩放百分比。

## 2. 架构总览

```text
┌─ Rust backend ────────────────────────────────────────┐
│                                                       │
│  pty              ─ 子进程 + PTY,sync + crossbeam     │
│  terminal-engine  ─ alacritty 适配,Snapshot 模型      │
│  terminal-protocol─ Init/Patch/Exited,RLE 行编码      │
│  terminal-input   ─ key/mouse/paste → PTY bytes       │
│  terminal-session ─ 缝合 4 层,events()/input() handle│
│                                                       │
│  perga-cli                                            │
│    ├─ bin "perga"        交互式 PTY demo (raw mode)   │
│    └─ bin "perga-bench"  sync 路径性能基准            │
│                                                       │
│  perga-server     ─ axum + tokio,WS /ws?rows=&cols=  │
│    bridge.rs:唯一 sync↔async 缝合点(OS 线程 +        │
│    crossbeam→tokio mpsc),严格保 PTY/engine 路径同步  │
│                                                       │
└───────────────────────────────────────────────────────┘
                       │ WebSocket (text JSON frames)
                       │ 协议:server→client = ProtocolEvent
                       │       client→server = ClientMessage
                       ▼
┌─ Web frontend (Vite + Solid + TS) ────────────────────┐
│                                                       │
│  state/protocol.ts    协议出站 TS 类型(与 Rust 对齐)│
│  state/wire.ts        协议入站 TS 类型                │
│  state/session.ts     纯函数 reducer + RLE 解码        │
│  state/session_store.ts  Solid createStore + produce  │
│  net/ws.ts            SessionSocket(单连接,无重连)│
│  render/metrics.ts    字体度量(hidden span 法)       │
│  render/palette.ts    Color → CSS(16色+256色 cube)   │
│  render/grid_canvas.tsx  Canvas 2D,run-grouping 优化 │
│  input/keyboard.ts    DOM KeyboardEvent → ClientMsg   │
│  input/resize.ts      ResizeObserver + 80ms debounce  │
│  ui/app.tsx           全屏单 session 装配              │
│  ui/perf_overlay.tsx  ?perf=1 启用,右上角实时指标   │
│  util/perf.ts         浏览器侧采样器                  │
│                                                       │
└───────────────────────────────────────────────────────┘
```

关键约束(出自 CLAUDE.md):

- PTY / engine / session 线程**严格 sync + crossbeam**,**不**进 tokio runtime。
- tokio 只在 `perga-server` 这一 crate 内作为 side pool。`bridge.rs` 是整个项目里唯一一处 sync 转 async 的位置。
- 协议数据先行(数据模型优先于特殊情况);不为未来写代码;边界验证;不静默吞错。

### 已完成 — Canvas run-grouping 优化

用户反馈 `dispatch p99 = 62ms`(在 release 后端 + release 前端下)。定位到 `drawCell` 每个 cell 都设 `fillStyle` 两次 + `font` 一次,80×24 = 1920 cells × 3 状态切换 = ~5760 canvas ops。

重写 `drawRow`:

- **run-grouping**:同 `(fg, bg, attrs)` 的连续单宽 cell 合并为一段 run,1 次 `fillRect` 画整段背景 + 1 次 `fillText` 画整段文字。
- **状态缓存**:`lastFillStyle` / `lastFont` 缓存,相同值不下发 setter。
- Wide char、cursor、wide_spacer 走单独的 `drawSingleCell` 回退路径。

期望 80 列全屏重绘从 ~400 canvas ops 降到 ~6-12 ops/行,dispatch p99 从 62ms 应降到 5-10ms 级别。**实际数字用户尚未回测**(见 §5)。

## 7. Phase 3 详细规划 — OSC 133 + Warp 风格命令块

**目标**:完成的命令收成可折叠 DOM block(可选中复制),当前正在输入 / 执行的命令仍是 inline Canvas grid 终端体验。

**OSC 133 协议**(shell integration 标准):

| 序列 | 含义 |
|---|---|
| `\e]133;A\e\\` | Prompt start |
| `\e]133;B\e\\` | Prompt end / command start(用户已经回车) |
| `\e]133;C\e\\` | Command execution start(命令开始运行) |
| `\e]133;D;<exit>\e\\` | Command finished,exit code |

需要用户在 shell 配置里 source 我们的 setup 脚本才会发这些序列。**不强制**:没 source 时降级到纯 Grid 模式,UX 平滑过渡。

**后端新文件**:

| 文件 | 内容 |
|---|---|
| `crates/terminal-engine/src/shell_integration.rs` | `ShellIntegration` 含一个 `vte::Parser` 旁路 + `Osc133Perform`,只 implement `osc_dispatch`。`ShellMark` 枚举:`PromptStart` / `CommandStart` / `CommandExecuted` / `CommandFinished { exit }` |
| `crates/terminal-engine/tests/osc133.rs` | 喂含 OSC 133 的字节,断言 marks 序列 |
| `crates/terminal-protocol/tests/marks.rs` | 三个新 ProtocolEvent 变体 round-trip serde |
| `scripts/perga-bash.sh` | `PROMPT_COMMAND` 钩 OSC 133 |
| `scripts/perga-zsh.sh` | `precmd` / `preexec` 钩 OSC 133 |

**后端修改**:

- `crates/terminal-engine/src/lib.rs`:`mod shell_integration` + re-export `ShellMark`。
- `crates/terminal-engine/src/engine.rs`:`TerminalEngine` 持 `ShellIntegration`,`feed()` 内部**双喂**(alacritty + 旁路 vte),新增 `drain_marks() -> Vec<ShellMark>`(对称 `drain_pending_writes`)。
- `crates/terminal-protocol/src/event.rs`:`ProtocolEvent` 加 `PromptStart { seq, row }` / `CommandStart { seq, row }` / `CommandEnd { seq, row, exit: Option<i32> }`。
- `crates/terminal-protocol/src/encoder.rs`:加 `encode_mark(ShellMark, viewport_row) -> ProtocolEvent`,seq 沿用单调计数器。
- `crates/terminal-session/src/event_loop.rs::handle_pty_event`:`engine.feed()` 后**先 drain marks emit,再 emit_frame**。⚠ 这个顺序关键 ── 否则 `CommandStart.row` 的视口行号会落在该 Patch 已应用之后,前端无法对齐。

**前端新文件**:

| 文件 | 内容 |
|---|---|
| `web/src/state/blocks.ts` | `CommandBlock = { id, exit, rows: RowEntry[][], capturedAt, startedAt }`,reducer 在 CommandEnd 时从 grid view-model 复制 `[commandStart.row, commandEnd.row]` 行段 |
| `web/src/render/block_list.tsx` | `<For>` 虚拟列表 DOM 渲染,foldable / 可选中复制 |
| `web/src/render/block_one.tsx` | 单个 block 的 DOM:命令行 header + 输出 body + exit 状态徽章 |

**前端修改**:

- `web/src/state/protocol.ts`:TS 加三个新 event variant。
- `web/src/state/session.ts`:`SessionState` 加 `marks` 字段(临时存当前进行中的 PromptStart / CommandStart row)+ `blocks: CommandBlock[]`(append-only),reducer 处理新事件。
- `web/src/state/session_store.ts`:`produce` 路径加 mark 处理。
- `web/src/ui/pane_leaf.tsx`(Phase 2 出的那个)→ 布局改成「上面 `<BlockList>` + 下面 `<GridCanvas>` 始终底部」。`alt_screen` 时挂起 BlockList 让 Canvas 占满。
- `README.md`:加 "Shell Setup" 段:`source scripts/perga-zsh.sh`(用户 opt-in)。

**设计要点**(已经定,不要反复纠结):

- OSC 133 用**旁路 `vte::Parser`**,**不**去 strip 原字节(alacritty 0.25 静默丢未注册 OSC,无副作用)。

- Block **不 reflow**:resize 时 block 保持原宽,过宽横向滚动。「数据模型优先 + 不为未来写代码」(`CLAUDE.md`)。
- Block ↔ Canvas **跨边界选择**:Phase 3 **不做**,延后到 Phase 7+。
- alt_screen 模式时 BlockList 暂停(vim / tmux 全屏占 Canvas)。
- 没 source shell 脚本的用户:看不到 block,但 Grid 一切正常工作 ── 降级路径就是 Phase 1 状态。

**验收**:

- `websocat` 能看到 `prompt_start` / `command_start` / `command_end` JSON 帧。
- 浏览器:跑 `echo hi` 看到 DOM block 出现,光标行依旧 inline 在 Canvas。
- vim 进 alt screen → BlockList 挂起;`:q` 退出 → 恢复。

## 9. Phase 5 详细规划 — SSH (russh) + host profile

| 文件 | 内容 |
|---|---|
| `src/lib.rs` | re-exports |

## 10. Phase 6 详细规划 — Tauri 打包

**目标**:把 Web 前端打包成桌面 app(Linux 优先 → Mac → Windows),然后 iPad / Android。

**新 crate**:`crates/perga-app`(Tauri 2 标准结构)。

**关键改动**:`web/src/net/ws.ts` 抽象出 `Transport` 接口,**两个实现**:

- `WebSocketTransport`:现有逻辑。
- `TauriIpcTransport`:用 `@tauri-apps/api` 的 `invoke` + `listen` 替代 WS 帧。Tauri 后端注册 `#[tauri::command] async fn send(...)` 和 emit event。

启动时 detect `window.__TAURI__` 存在与否,自动选 transport。**前端业务代码完全不感知**这层差异。

**Tauri 后端**:

| 文件 | 内容 |
|---|---|
| `crates/perga-app/src-tauri/src/main.rs` | Tauri 入口,注册 commands,启 PTY/engine/session 同一套核心(直接 import 现有 5 个 crate) |
| `crates/perga-app/src-tauri/tauri.conf.json` | bundle 配置 |

**移动版**(iPad / Android):**Tauri 2 mobile 当前状态决定**。如果 mobile 还不稳定,这部分延期。需要查最新 Tauri 2 mobile 文档。

平板专属问题:

- 触屏 split / 关闭手势。
- 虚拟键盘弹起时不能挡终端。
- 横屏 ↔ 竖屏切换 → resize。
- 键盘特殊键(Esc / Ctrl)在 iOS 软键盘上没有 → 需要 UI 提供。

## 12. 设计决策清单(都已和用户对齐)

跨会话需要记住、容易反复纠结的几个决定:

| 议题 | 决定 | 理由 |
|---|---|---|
| Tauri vs 纯 Web | **纯 Web 先做,Tauri 后置** | Tauri 调试摩擦大;真正问题是 transport 形态,Tauri 后期只是把 WebSocket 换 Tauri IPC,前端代码不动 |
| HTTP control plane | **不做**,只 `/ws?rows=&cols=` | Phase 0-4 都没 reconnect / session listing 需求,加了就是死代码 |
| 多 WS 多路复用 | **不做**,一会话一连接 | tabs / panes 是前端概念,每个 leaf 自己开一条 WS,server 无状态 |
| SSH 实现 | **deferred 到 Phase 5+,届时用 russh** | iPad 沙箱不允许 spawn `ssh` 子进程;桌面也跟着 russh 统一一套 |
| 远程访问 auth | **localhost-only bind,部署阶段用 Tailscale** | 不写鉴权代码 |
| 前端框架 | **Solid** | Signals 细粒度更新最契合「dirty-row 高频小 patch」;用户不会前端,Solid API 易学 |
| Grid 渲染 | **Canvas 2D**,Block 渲染用 DOM | Canvas 对终端 redraw 性能上限高;Block 需要选择/复制/折叠 → DOM 是正解 |
| OSC 133 接入 | **vte 旁路 Parser**,双喂同一份字节 | alacritty 0.25 不暴露 OSC 133,vte 默认丢弃;不 strip(alacritty 静默丢) |
| 协议事件序 | **`drain_marks()` 先于 `emit_frame`** | `CommandStart.row` 必须在该 Patch 应用前采样,否则前端对齐不上 |
| Block 不 reflow | resize 时 Block 保持原宽度,过宽横向滚动 | 「不为未来写代码」,真要 reflow 等用户提需求 |
| WS 重连 | **故意不做** | Reconnect ≠ 接到原 session(WS 一断后端 PTY 就死);重连等同新开会话,UX 上必须明示 |

## 13. 已知风险 / 待回答的问题

- **性能天花板**:Web 终端架构上有一个 ~33ms 的输入→显示延迟下限(浏览器合成 + 显示器刷新),Wezterm 走 GPU 原生约 8-16ms。改不掉。用户的延迟感是不是 30ms 这个量级,需要 perf overlay 数字回来才能判断 ── 现在最大悬念。
- **PTY 子进程清理**:`TerminalSession::Drop` 已经覆盖了 SIGHUP→500ms→SIGKILL 序列,terminal-session 测试覆盖。但 perga-server 在 server 进程异常退出时(panic 非 catch_unwind 路径)是否能保证清掉所有 PTY?需要 Phase 2 加 session registry 时一并审。
- **OSC 133 时 `row` 字段采样窗口**:Plan 里设计是 mark 触发瞬间的视口行号,但 alacritty 是黑盒 ── 真实抓 vt100 序列时 cursor 可能还没移到 prompt 行。Phase 3 实测验证。
- **Solid `produce` 的反应粒度**:目前假设 `draft.grid[idx] = newRow` 触发 grid[idx] 单点失效,reactive 跟踪粒度落到 row 级。**未实证**;如果实测发现整个 grid 失效(导致整屏重绘),需要换 setStore 路径选择器(`setState('grid', idx, ...)`)。
