## 1. 总体目标

Perga 是一个**以 Web 前端为主**、Rust 后端接管 PTY 和终端协议解析的新形态终端。终端字节流由后端 alacritty 内核解析,前端只消费规范化后的协议事件(`Init` / `Patch` / `Exited`)。

最终形态(目标态):

- 桌面 Tauri app:本地 PTY + russh,所有 auth 本地。
- iPad / Android Tauri app:russh 直连远程机器(平板沙箱不允许 spawn ssh 子进程)。
- 中间形态(当前开发态):**纯 Web,前端浏览器 ↔ 本机 Rust server**,绕开 Tauri 调试摩擦。

用户给出的 5 个产品需求:

1. 好看,自定义布局。
2. 多 SSH 会话。
3. **未来重要场景是 iPad / Android 远程 SSH 连一台机器**。这反过来要求 SSH 走 russh,不走 `ssh` 子进程。
4. 分块布局 = **tabs(顶部)+ panes(tab 内分屏)+ Warp 风格命令 block(完成的命令收成 DOM 块,但当前行仍是 inline 终端体验,不是固定底部输入框)**。
5. 自定义缩放百分比。

## 2. 架构总览

```text
┌─ Rust backend ────────────────────────────────────────┐
│                                                       │
│  pty              ─ 子进程 + PTY,sync + crossbeam     │
│  terminal-engine  ─ alacritty 适配,Snapshot 模型      │
│  terminal-protocol─ Init/Patch/Exited,RLE 行编码      │
│  terminal-input   ─ key/mouse/paste → PTY bytes       │
│  terminal-session ─ 缝合 4 层,events()/input() handle│
│                                                       │
│  perga-cli                                            │
│    ├─ bin "perga"        交互式 PTY demo (raw mode)   │
│    └─ bin "perga-bench"  sync 路径性能基准            │
│                                                       │
│  perga-server     ─ axum + tokio,WS /ws?rows=&cols=  │
│    bridge.rs:唯一 sync↔async 缝合点(OS 线程 +        │
│    crossbeam→tokio mpsc),严格保 PTY/engine 路径同步  │
│                                                       │
└───────────────────────────────────────────────────────┘
                       │ WebSocket (text JSON frames)
                       │ 协议:server→client = ProtocolEvent
                       │       client→server = ClientMessage
                       ▼
┌─ Web frontend (Vite + Solid + TS) ────────────────────┐
│                                                       │
│  state/protocol.ts    协议出站 TS 类型(与 Rust 对齐)│
│  state/wire.ts        协议入站 TS 类型                │
│  state/session.ts     纯函数 reducer + RLE 解码        │
│  state/session_store.ts  Solid createStore + produce  │
│  net/ws.ts            SessionSocket(单连接,无重连)│
│  render/metrics.ts    字体度量(hidden span 法)       │
│  render/palette.ts    Color → CSS(16色+256色 cube)   │
│  render/grid_canvas.tsx  Canvas 2D,run-grouping 优化 │
│  input/keyboard.ts    DOM KeyboardEvent → ClientMsg   │
│  input/resize.ts      ResizeObserver + 80ms debounce  │
│  ui/app.tsx           全屏单 session 装配              │
│  ui/perf_overlay.tsx  ?perf=1 启用,右上角实时指标   │
│  util/perf.ts         浏览器侧采样器                  │
│                                                       │
└───────────────────────────────────────────────────────┘
```

关键约束(出自 CLAUDE.md):

- PTY / engine / session 线程**严格 sync + crossbeam**,**不**进 tokio runtime。
- tokio 只在 `perga-server` 这一 crate 内作为 side pool。`bridge.rs` 是整个项目里唯一一处 sync 转 async 的位置。
- 协议数据先行(数据模型优先于特殊情况);不为未来写代码;边界验证;不静默吞错。

### 已完成 — Canvas run-grouping 优化

用户反馈 `dispatch p99 = 62ms`(在 release 后端 + release 前端下)。定位到 `drawCell` 每个 cell 都设 `fillStyle` 两次 + `font` 一次,80×24 = 1920 cells × 3 状态切换 = ~5760 canvas ops。

重写 `drawRow`:

- **run-grouping**:同 `(fg, bg, attrs)` 的连续单宽 cell 合并为一段 run,1 次 `fillRect` 画整段背景 + 1 次 `fillText` 画整段文字。
- **状态缓存**:`lastFillStyle` / `lastFont` 缓存,相同值不下发 setter。
- Wide char、cursor、wide_spacer 走单独的 `drawSingleCell` 回退路径。

期望 80 列全屏重绘从 ~400 canvas ops 降到 ~6-12 ops/行。run-grouping 压低了每行成本,但当时没发现「每个 patch 都在全屏重绘」这个更上游的 bug(见下一节)。

### 已完成 — 前端 dispatch 性能优化(本轮)

run-grouping 后 dispatch p99 仍有 44ms。定位到三层根因并逐一修掉:

1. **`syncCanvasSize` 比较 bug**:`grid_canvas.tsx` 的 canvas 尺寸检查写成
   `Math.round(X) !== X`,对任意非整数 X 恒为真 → **每个 patch 都全屏重绘**、
   脏行缓存被清空,增量重绘形同虚设。修成两边都比 round 后的目标值。
2. **grid 移出 Solid store**:`SessionState.grid` 拆成 store 外的 raw `Cell[][]`
   backing buffer,渲染热路径不再穿 Solid proxy。
3. **RAF 合批**:`grid_canvas.tsx` 把脏行收进 `pendingRows`,合并到一帧 RAF 统一画。

`ls` ×100 autotest 回测:

| 指标 | 改造前 | 现在 |
|---|---|---|
| dispatch p99 | 43.8 ms | **0.4 ms** |
| dispatch p50 | 16.6 ms | **0.1 ms** |
| latency enter→settle p99 | 143.9 ms | **4.3 ms** |
| render canvas RAF p99 | (并入 dispatch) | 1.1 ms |

大幅超过「p99 < 10ms」目标。原计划的后端 coalescing / 协议 scroll op 触发条件是
「p99 仍 > 10ms」,未命中 → 不做(不为未来写代码)。

> ⚠️ **命令块系统已于 2026-05-23 整体移除**(见 §5 末节)。本节是历史规划,

> 和 §5 / §12 / §13 为准。
>
> 🔬 **探测修订(2026-05-21)**:OSC 133 落地前先跑了一次纯探测
> (`crates/perga-cli/src/bin/probe_osc133.rs` + `scripts/perga-bash.sh`),
> 起真实 bash 实测 mark 行号语义,结论见 §13。两点改了本节原规划:
> ① `feed()` 双喂必须在 OSC 边界**交错**推进两个状态机,不能整块喂完再喂;
> ② block 内容由**后端**在 CommandEnd 组装下发,前端不再切自己的视口
> —— 视口行号会因滚动失效。下方已就地修订。

**目标**:完成的命令收成可折叠 DOM block(可选中复制),当前正在输入 / 执行的命令仍是 inline Canvas grid 终端体验。

**OSC 133 协议**(shell integration 标准):

| 序列 | 含义 |
|---|---|
| `\e]133;A\e\\` | Prompt start |
| `\e]133;B\e\\` | Prompt end / command start(用户已经回车) |
| `\e]133;C\e\\` | Command execution start(命令开始运行) |
| `\e]133;D;<exit>\e\\` | Command finished,exit code |

需要用户在 shell 配置里 source 我们的 setup 脚本才会发这些序列。**不强制**:没 source 时降级到纯 Grid 模式,UX 平滑过渡。

**后端新文件**:

| `crates/terminal-engine/tests/osc133.rs` | 喂含 OSC 133 的字节,断言 marks 序列 |
| `crates/terminal-protocol/tests/marks.rs` | 三个新 ProtocolEvent 变体 round-trip serde |

| `scripts/perga-zsh.sh` | `precmd` / `preexec` 钩 OSC 133 |

**后端修改**:

- `crates/terminal-engine/src/lib.rs`:`mod shell_integration` + re-export `ShellMark`。
- `crates/terminal-engine/src/engine.rs`:`TerminalEngine` 持 `ShellIntegration`,`feed()` 内部**双喂**(alacritty + 旁路 vte)。⚠ 双喂必须在 OSC 边界**交错**推进两个状态机 —— 整块喂完 alacritty 再喂 vte,会让 mark 的光标采样落到 chunk 末尾、行号全错(探测已证实)。`ShellMark` 在旁路 parser 命中那一刻立即从 alacritty 抓**绝对行号**。新增 `drain_marks() -> Vec<ShellMark>`(对称 `drain_pending_writes`)。
- `crates/terminal-protocol/src/event.rs`:`ProtocolEvent` 加 `PromptStart { seq, line }` / `CommandStart { seq, line }` / `CommandEnd { seq, exit: Option<i32>, rows }`。`line` 是 alacritty 的**绝对行号**(不随滚动失效),不是视口行;`CommandEnd.rows` 直接携带后端组装好的 block 行段(复用既有 RLE `RowEntry` 编码),前端不再自己切视口。
- `crates/terminal-protocol/src/encoder.rs`:加 `encode_mark(ShellMark) -> ProtocolEvent`;收到 `D` mark 时,后端从 grid(含 scrollback)按绝对行取 `[CommandStart.line, CommandEnd.line)` 组装 `CommandEnd.rows`。seq 沿用单调计数器。
- `crates/terminal-session/src/event_loop.rs::handle_pty_event`:`engine.feed()` 后**先 drain marks emit,再 emit_frame**。绝对行号本身不怕 Patch 乱序,但 emit 顺序仍要保证 —— `CommandEnd`(带 block 行段)必须在对应 Patch 之前到,前端才能在正确时机把活动区收成块。

|---|---|
| `web/src/state/blocks.ts` | `CommandBlock = { id, exit, rows: RowEntry[][], capturedAt, startedAt }`,reducer 在 `CommandEnd` 时直接收下后端组装好的 `rows`,**不**自己切视口 —— 视口只有一屏,超一屏的命令切不全(探测已证实) |

**设计要点**(已经定,不要反复纠结):

- OSC 133 用**旁路 `vte::Parser`**,**不**去 strip 原字节(alacritty 0.25 静默丢未注册 OSC,引擎状态零污染 —— 探测已确认)。

| Mouse reporting | 后端 wire/session/input encoder 已实现,前端只有 TS wire 类型,还没绑 pointer/wheel 发送。复制选择先默认接管鼠标;后续接 mouse reporting 后,`mouse_reporting !== off` 时默认把鼠标交给 TUI,`Shift/Alt+Drag` 强制前端选择 | 用户要 tmux/vim/less/htop/lazygit 鼠标操作 |
| IME 输入 | `compositionstart` / `compositionend`,合成完整字符串后整段发 paste 帧 | 用户 CN 输入需要 |
| 主题自定义 | TOML 配置文件,用户可定义 16 色 palette + 字体 | Phase 4 之后 |
| Scrollback 持久化 | session 关闭后保留输出记录 | 用户报告需要 |
| 服务端 session listing | 跨 window 多连接接同一 session | 想做 reconnect 时考虑 |
| WS 重连 | 故意不做(reconnect ≠ 接到原 session)── 但 mobile 网络抖动可能强制要求 | Phase 6 mobile 时评估 |

**横向问题**(任何阶段都可能撞上):

## 12. 设计决策清单(都已和用户对齐)

跨会话需要记住、容易反复纠结的几个决定:

| 议题 | 决定 | 理由 |
|---|---|---|
| Tauri vs 纯 Web | **纯 Web 先做,Tauri 后置** | Tauri 调试摩擦大;真正问题是 transport 形态,Tauri 后期只是把 WebSocket 换 Tauri IPC,前端代码不动 |
| HTTP control plane | **不做**,只 `/ws?rows=&cols=` | Phase 0-4 都没 reconnect / session listing 需求,加了就是死代码 |
| 多 WS 多路复用 | **不做**,一会话一连接 | tabs / panes 是前端概念,每个 leaf 自己开一条 WS,server 无状态 |
| SSH 实现 | **deferred 到 Phase 5+,届时用 russh** | iPad 沙箱不允许 spawn `ssh` 子进程;桌面也跟着 russh 统一一套 |
| 远程访问 auth | **localhost-only bind,部署阶段用 Tailscale** | 不写鉴权代码 |
| 前端框架 | **Solid** | Signals 细粒度更新最契合「dirty-row 高频小 patch」;用户不会前端,Solid API 易学 |
| Grid / scrollback 渲染 | **活动区 Canvas 2D,历史区虚拟 DOM 行** | Canvas 对终端 redraw 性能上限高;历史需要选择/复制与大列表虚拟化 → DOM 行更合适 |
| OSC 133 接入 | **vte 旁路 Parser**,双喂、按 OSC 边界交错 | alacritty 0.25 不暴露 OSC 133,vte 默认丢弃;不 strip(alacritty 静默丢);交错喂才能采到正确 mark 行 |
| bash 集成 `C` 钩子 | **走 `PS0`,不走 `DEBUG` trap** | DEBUG trap 会被 PROMPT_COMMAND 每条命令、PS1 命令替换乱触发,和用户已有的 starship / direnv 冲突;`PS0` 是「命令执行前」的独立钩子,与 PROMPT_COMMAND 无关,干净(bash 4.4+) |
| 协议事件序 | **`command_end` 先于对应 patch** | 前端要先记录失败命令行号,随后 patch / scrolled_rows 把该行滚进 history 时才能落红色失败标记 |
| marks 上不上 wire | **只 `command_end` 上 wire** | prompt / command-start 仍是后端内部状态;前端当前只需要失败命令的 exit + line,回显 settle 不值得单开事件 |
| scrollback wire | **Patch 携带 `scrolled_rows` / `cleared`,不再有 `active_top`** | 命令块移除后 Canvas 画完整 viewport;滚出 viewport 的历史由前端累积,`CSI 3J` 通过 `cleared` 清 history |
| resize / alt-screen 重新基准化 | **`scroll_total` 归零、丢在途 command region** | reflow / alt-screen 会让绝对行号语义失真;代价是跨切换的那条失败标记可能丢失,罕见可接受 |
| WS 重连 | **故意不做** | Reconnect ≠ 接到原 session(WS 一断后端 PTY 就死);重连等同新开会话,UX 上必须明示 |
| 复制 vs TUI 鼠标 | **前端选择系统要尊重 `mouse_reporting`** | 现阶段前端还没发 mouse 帧,可默认拖拽选择;后续 TUI 鼠标接入后,mouse reporting 开启时默认鼠标交给应用,用 `Shift/Alt+Drag` 强制选择文本 |

## 13. 已知风险 / 待回答的问题

- **性能天花板**(已回测):前端处理已不是瓶颈 ── dispatch p99 0.4ms、latency enter→settle p99 4.3ms。剩下的是 Web 架构固有的 ~33ms 输入→显示下限(浏览器合成 + 显示器刷新),Wezterm 走 GPU 原生约 8-16ms,改不掉,接受。
- **PTY 子进程清理**:`TerminalSession::Drop` 已经覆盖了 SIGHUP→500ms→SIGKILL 序列,terminal-session 测试覆盖。但 perga-server 在 server 进程异常退出时(panic 非 catch_unwind 路径)是否能保证清掉所有 PTY?需要 Phase 2 加 session registry 时一并审。
- **OSC 133 行号语义**(✅ 已落地):探测确认 mark 的语义行能精确采到
  (A/B 在 prompt 行、C 在输出首行、D 在末输出行下一行),但视口行号会因滚动
  失效。Phase 4R 后命令块已移除,OSC 133 只用于 `command_end { exit,line }`
  失败标记;line 仍靠后端绝对行号。
  探测器 `probe_osc133.rs` 已随 Phase 3 收尾删除,回归由
  `crates/terminal-engine/tests/osc133.rs` 接管。
- **scrollback 饱和**:绝对行号 / `scrolled_rows` 仍靠 alacritty
  `history_size()` 增量累计;scrollback 上限 10000 行,饱和后增量恒 0、历史停止
  增长、失败标记行号也会漂移。详见 `TODO.md`。
